## ⚠️ Key findings to double-check before migrating

1. **Grant drift (anon has SELECT on newer tables).** Migrations only `GRANT … TO authenticated`/`service_role`, but the live DB also grants **`anon` SELECT** on every table created *after* the June‑17 lockdown migration: `customers`, `jobs`, `companies`, `user_roles`, `trade_presets`, `email_send_log`, `email_send_state`, `suppressed_emails`, `email_unsubscribe_tokens`. This is almost certainly Supabase **default privileges** (`ALTER DEFAULT PRIVILEGES … GRANT … TO anon`) that are **not represented in any migration file**.
   - **No data is leaking** — every one of these returns `[]` because RLS is enabled with no anon-facing policy. But the grant is broader than the migrations imply, and RLS is the *only* thing holding it back. Verify default privileges in the source project and decide whether to replicate them.
   - The older, explicitly-revoked tables correctly return `permission denied`: `profiles`, `agent_rules`, `neighbor_outreach`, `team_members`, `job_assignments`, `job_locks`.

2. **Cascade-dropped foreign keys never re-added.** Migration `20260618142846` ran `DROP TABLE public.jobs CASCADE` (and `customers CASCADE`). That cascade silently dropped the FKs pointing at `jobs` from **`neighbor_outreach.job_id`**, **`job_assignments.job_id`**, and **`job_locks.job_id`** (and `jobs.scheduled_by_id → team_members`). The recreated `jobs` table did **not** restore them. `types.ts` confirms these relationships are now absent. On the live DB these columns are **unconstrained `uuid`** — orphan rows are possible. Recreate intentionally (or deliberately omit) in the new project.

3. **Objects applied dynamically — NOT in static SQL.** The email migration's footer documents steps applied at runtime via the Supabase Management API, so `supabase db push` against a fresh project will **not** create them:
   - **pg_cron job** `process-email-queue` (5-second interval).
   - **Vault secret** `email_queue_service_role_key`.
   - **pgmq queues** `auth_emails`, `transactional_emails`, `auth_emails_dlq`, `transactional_emails_dlq` (created via `pgmq.create()` in a `DO` block — these *are* in SQL, but depend on the `pgmq` extension being present first).

4. **`team_members.company_id` is `text`, not `uuid`** (default `'vantage-co'`), unlike every other tenant table. RLS casts `current_company_id()::text` to compare. Intentional legacy quirk — carry it over as-is or normalize during migration.

5. **`jobs.status` and `customers.customer_type` are plain `text`**, even though `job_status` and `customer_type` enums still exist (the original enum-typed columns were dropped with the table recreate). The enums are now effectively unused by those columns.

## Could NOT be verified directly — confirm manually before/after migration

The anon key cannot read system catalogs, so everything below should be confirmed via the Supabase dashboard SQL editor or `supabase db pull` against the source project:

1. **That the live schema exactly matches the migrations** (no out-of-band changes made via the Lovable UI). Run `supabase db pull` / `pg_dump --schema-only` from the source for the ground truth.
2. **Exact numeric precision/scale.** Migrations use bare `numeric` for money columns on the recreated `jobs`/`customers` and elsewhere (the original `jobs.quote_amount` was `numeric(12,2)` before recreate). Confirm precision in the live catalog.
3. **RLS `enabled` flag per table** — strongly inferred from empty results + permission errors, but not introspected.
4. **Existence/exact bodies of all triggers, functions, and indexes** listed above.
5. **The grant matrix** — especially the `anon` default-privilege SELECTs (finding #1) and whether `ALTER DEFAULT PRIVILEGES` is set in the source.
6. **Extensions, pgmq queues, the pg_cron job, and the vault secret** (finding #3) — these will not transfer by SQL migration alone.
7. **`auth` schema:** users, identities, providers (Google/Apple OAuth), email templates, auth settings, JWT/SMTP config — none of this is in these migrations.
8. **Storage buckets / objects**, **Edge Functions** (e.g. `process-email-queue`, suppression handler), and their **secrets/env**.
9. **Actual data volume / row counts** — every tenant table returned `[]` to anon (RLS), so I have no read on real row counts.
10. **Any cron jobs, webhooks, or database settings** configured outside migrations.
